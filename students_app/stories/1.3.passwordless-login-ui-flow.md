# Story 1.3: The Passwordless Login UI & Flow

## Status

Draft

## Story

**As a** Student (Camper),
**I want** to log in using only my email address and a 6-digit code,
**so that** I don't have to remember a password during the workshop.

## Acceptance Criteria

1. A `/login` page exists within the `/(camper)` group (specifically `/(camper)/camper/login`) featuring the Epitech design system (dark mode aesthetic, technical grids).
2. The student enters their email. If the email exists in the `students` DB, the system generates a 6-digit OTP and sends an email.
3. The UI smoothly transitions to an "Enter Code" screen without full page reloads.
4. Upon entering the correct code before it expires, the system authenticates the user, sets the `pb_student_auth` cookie, and redirects to the Camper Dashboard (`/(camper)/camper`).
5. Invalid or expired codes display a clear error message.

## Tasks / Subtasks

- [ ] Create `/(camper)/camper/login/+page.server.ts` for form actions.
  - [ ] Implement `requestOtp` action: Validate email, call PocketBase `requestOTP(email)` on the `students` collection, and return the resulting `otpId`.
  - [ ] Implement `verifyOtp` action: Accept the 6-digit code and `otpId`, call PocketBase `authWithOTP(otpId, code)`, and handle errors.
  - [ ] On successful `verifyOtp`, export the `locals.studentPb.authStore` to a cookie (`pb_student_auth`) and append it to `response.headers`.
  - [ ] Redirect the user to `/(camper)/camper` on success.
- [ ] Implement `/(camper)/camper/login/+page.svelte` UI.
  - [ ] Use `sveltekit-superforms` to handle the dual-step form process (Email step -> OTP step). -[ ] Build the "Step 1: Email Input" UI.
  - [ ] Build the "Step 2: Enter Code" UI (displayed conditionally when an `otpId` is present).
  - [ ] Style the login page to match the "Cockpit" aesthetic (incorporate existing `layout.css` variables, Anton/IBM Plex Sans fonts, and background grids similar to `admin/login`).
  - [ ] Implement clear, styled error handling using the existing `Alert` components for invalid emails or wrong OTP codes.

## Dev Notes

- **PocketBase OTP API**: PocketBase v0.23+ natively supports OTP for Auth collections. You do not need a custom `otps` collection if using the built-in feature.
  - To request a code: `await locals.studentPb.collection('students').requestOTP(email)` -> returns an object containing an `otpId`.
  - To verify a code: `await locals.studentPb.collection('students').authWithOTP(otpId, code)`.
- **Cookie Handling**: Look at `hooks.server.ts`. It's already configured to extract and inject `pb_student_auth` into `locals.studentPb`. You just need to ensure that after `authWithOTP` succeeds, you set the response cookie exactly like the other auth pages do (e.g., using `studentPb.authStore.exportToCookie(cookieOpts)`).
- **Security/UX Practice**: When requesting an OTP, if the email does not exist in the database, PocketBase will throw a 404. To prevent email enumeration, catch this error and return a generic success message to the client anyway ("If an account exists, a code has been sent"). However, for the MVP and camper context, it's often more user-friendly to just tell them "Email not found" so they don't wait for an email that will never arrive. Follow standard Superforms error messaging.
- **Styling Reference**: Reference `src/routes/(auth)/login/+page.svelte` and `src/routes/(auth)/admin/login/+page.svelte` for the Epitech grid background, vignette overlays, and Card styling.

## Testing

- **Test File Location**: No explicit unit tests required for this UI story, but manual flow testing is critical.
- **Test Standards**:
  - Verify email input triggers an actual email from the local/dev PocketBase instance (check PocketBase admin UI logs/mail dev catcher).
  - Verify entering an incorrect code throws a validation error.
  - Verify a correct code logs the user in and redirects to `/(camper)/camper`.
  - Verify that a logged-in student navigating back to `/camper/login` gets automatically redirected to `/camper`. (This guard should already be active from Story 1.2 in `hooks.server.ts`).
